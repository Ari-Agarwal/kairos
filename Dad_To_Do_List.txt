1) BUY A CUSTOM DOMAIN
-----------------------
Why: Right now the app lives at "telos-zeta.vercel.app". That address is
owned by Vercel (the hosting company), not us, so we cannot add the special
DNS records needed to send email reliably or to put a real ".com" address
on the app. We need to own our own domain name first.

Steps:
  a. Go to a domain registrar website. Recommended: Namecheap.com or
     Cloudflare.com (Cloudflare is usually cheapest, no markup).
  b. Search for a domain, e.g. "telos.com", "telosapp.com", "gettelos.com",
     "usetelos.com" — try a few until you find one that's available.
     Cost is usually $10-15 for the first year for a ".com".
  c. Dad pays with his card and completes checkout.
  d. After purchase, log in to the registrar's dashboard. You'll land on a
     page for "Domain List" or "My Domains" — find the domain you just
     bought and click into its management/DNS settings.
  e. Keep this dashboard open/bookmarked — you'll come back to it for
     steps 2 and 3.
  f. Tell me (Claude) the exact domain name you bought once it's done.


2) CONNECT THE DOMAIN TO VERCEL (so the app loads at the new address)
-----------------------------------------------------------------------
Why: Buying the domain doesn't automatically point it at our app. We have
to tell Vercel "this domain belongs to our project" and tell the registrar
"send traffic for this domain to Vercel."

Steps:
  a. Go to vercel.com and log in to the account the Telos project is
     deployed under.
  b. Open the Telos project, click "Settings" in the top nav, then
     "Domains" in the left sidebar.
  c. Type in the new domain name (e.g. "usetelos.com") and click "Add".
  d. Vercel will show you one or two DNS records to add — usually:
       - An "A" record pointing to an IP address (something like
         76.76.21.21), OR
       - A "CNAME" record pointing to "cname.vercel-dns.com"
     Vercel shows you exactly which ones on screen — take a screenshot or
     write them down.
  e. Go back to the registrar dashboard from step 1, find "DNS Settings"
     or "Manage DNS" for the domain.
  f. Add the exact record(s) Vercel showed you. Most registrars have an
     "Add Record" button — choose the record type (A or CNAME), paste in
     the host/name and the value Vercel gave you, and save.
  g. Wait. DNS changes can take anywhere from 10 minutes to 48 hours to
     "propagate" (take effect everywhere on the internet). Vercel's
     Domains page will show a green checkmark once it's verified — just
     check back on it periodically.
  h. Once verified, typing the new domain into a browser should load the
     Telos app.


3) SET UP EMAIL DELIVERABILITY (SPF / DKIM / DMARC)
------------------------------------------------------
Why: We use a service called Resend to send the welcome email when someone
signs up. Right now it sends from a generic Resend address
("onboarding@resend.dev") because we don't have our own domain. Once we
have a domain, we need to add three special DNS records so that email
providers (Gmail, Outlook, etc.) trust that our emails are really from us
and don't dump them in spam.

Steps:
  a. Go to resend.com and log in (or Dad/you create a free account if one
     doesn't exist yet — Resend's free tier covers our needs).
  b. In the Resend dashboard, find "Domains" in the left sidebar and click
     "Add Domain".
  c. Type in the new domain (e.g. "usetelos.com") and click through.
  d. Resend will display 3-4 DNS records to add — they'll be labeled
     something like:
       - TXT record for SPF
       - TXT or CNAME records for DKIM (often 2-3 of these)
       - TXT record for DMARC
     Resend shows the exact "Type", "Name/Host", and "Value" for each —
     write these down or keep the tab open.
  e. Go back to the registrar's "DNS Settings" page (same place as step 2).
  f. Add each record one at a time: pick the matching type (TXT or CNAME),
     paste in the exact name/host and value Resend gave you, save.
     Important: copy these exactly — a single typo will make verification
     fail.
  g. Go back to the Resend dashboard and click "Verify DNS Records" (or it
     may auto-check). This can also take a few hours to fully propagate,
     same as step 2.
  h. Once verified, tell me — I'll update the code (the EMAIL_FROM setting)
     to send from something like "noreply@usetelos.com" instead of the
     Resend sandbox address. This is a code change I can make myself once
     the domain is verified; you won't need Dad for that last part.


4) FUND THE ANTHROPIC (AI) API BILLING ACCOUNT
----------------------------------------------------------------
Why: Every time a student generates school matches, a timeline, essay
feedback, or a career path explanation, the app calls Anthropic's Claude
API to generate that content. Anthropic charges per "token" (roughly per
word) of text sent and received. This is metered, pay-as-you-go usage —
not a flat subscription — and it requires a credit card on file or
prepaid credits. If the account runs out of credits or has no payment
method, every one of those 4 AI features will start failing in production
for real students. This is required before/at launch, not optional.

Steps:
  a. Go to console.anthropic.com and log in to the account that holds our
     production API key (this should be the same Anthropic account from
     the Day 1 2FA setup).
  b. Click "Settings" (or "Billing") in the left sidebar, then "Billing"
     / "Plans & Billing".
  c. Add a payment method (credit card) if one isn't already on file.
  d. Either: (1) enable auto-reload so it automatically charges the card
     and adds credits when the balance gets low, or (2) manually purchase
     a starting credit balance — $20-50 is a reasonable amount to start
     with for a small number of real users; we can watch usage after
     launch and increase if needed.
  e. Once a payment method/credits are added, double check the "Usage"
     or "Limits" page so we know the current rate limit isn't capped too
     low for real traffic (the default limits are usually fine to start).
  f. Tell me once this is done — no code changes needed on my end, the
     existing API key keeps working, it just stops being blocked by a
     zero balance.
