### 2. Other website improvement ideas (not yet scheduled — brainstormed Jul 17)

Broader product ideas surfaced while synthesizing the focus group, beyond the direct action items above. These aren't committed work — flagging for prioritization discussion.

- **Re-organize the sidebar around the "3 jobs" students actually described** (roadmap/timeline, narrowing schools, essay support) rather than a flat feature list — e.g. group nav under "Plan," "Explore Schools," "Apply," mirroring the mental model that came up repeatedly across all four participants.
- **Make the "why this school" rationale more prominent on the Matches list itself**, not just inside each school's detail tabs — participants wanted to see reasoning up front when narrowing down a list, not after clicking in.
- **A lightweight "compare schools side-by-side" view** — several factor lists (weather, size, program, cost, community) are inherently comparative; a 2-3 school comparison table could reduce the "I don't know what I want" paralysis Caroline described.
- **Progress/momentum indicators on the dashboard** — a simple "how far along you are in your journey" visual (mentioned explicitly in the paper notes under Timeline) beyond the existing timeline list — e.g. a percent-complete or milestone tracker on the home screen.
- **Essay tool: show a visible "AI contribution" boundary in the UI itself** (not just copy) — e.g. clearly separating "your draft" from "AI feedback" panes so the product visibly reinforces that the student is the author, addressing the detection-fear trust gap structurally, not just with a disclaimer line.
- **A short "what colleges are looking for" explainer woven into onboarding or the school-list screen** — Caroline specifically flagged not knowing what she wants because she's only seen big schools; a brief comparative primer (urban vs. rural, size tradeoffs, etc.) alongside the factor-picker could reduce that specific confusion.
- **Revisit the mock interview and activity evaluation features' discoverability** — both were named in the session's top-3 lists but are currently a few clicks deep in nav; consider surfacing one of them from the dashboard directly.
- **A student-facing "trust/methodology" link placed near matches and timeline output**, not just a standalone `/methodology` page — put the trust-building content where the skepticism actually shows up (right next to the AI-generated recommendation), rather than requiring a separate visit.

### 7. Native app store development (Jul 21, flagged as a priority — PWA isn't cutting it)

Explicit call-out: a PWA/responsive-web experience isn't enough; Kairos needs real App Store (iOS) and Play Store (Android) presence. This is a materially bigger scope than anything else on this list (new build/release pipeline, not just new screens) so it's broken out on its own rather than folded into 6h.

- **Framework decision** — the app is Next.js/React web today; the practical options are (a) React Native / Expo sharing logic and possibly some UI patterns with the existing React codebase, (b) wrapping the existing web app in a native shell (Capacitor/Tauri-style) for faster time-to-store at the cost of feeling less native, or (c) a fully separate native codebase (Swift/Kotlin) for the best platform feel at the highest cost/maintenance burden. Given team size (appears to be a single founder-engineer per the timeline history), Expo/React Native is the pragmatic default — reuses TypeScript logic and the team's existing React fluency, avoids maintaining two native codebases, and has a mature path to both stores from one codebase.
- **What ports cleanly vs. needs rethinking**: read-heavy screens (matches, timeline, scholarships list, career path) port fairly directly; anything relying on browser-only APIs needs native equivalents — the mock interview's Web Speech API (`onresult` handler, Section 4) has no direct RN equivalent and would need a native speech-to-text package; `.ics` calendar export/sync (Section 1) maps to native calendar integration (`expo-calendar` or platform-native calendar APIs) instead of file download links; SMS nudges are already server-side and platform-agnostic, but should be paired with real **push notifications** once native, which are a much stronger engagement channel than SMS/email for a teen audience (ties into 6b's re-engagement ideas — push is the natural upgrade path once the app exists).
- **Auth continuity** — confirm whatever auth provider Supabase is using has a clean mobile SDK/deep-link flow so a student isn't forced to re-onboard on mobile after onboarding on web (or vice versa); session/account should feel like one product across web and app, not two.
- **Offline/poor-connectivity behavior** — unlike a desktop web session, a student's phone will genuinely go offline (bad reception, school wifi restrictions); worth deciding what should work offline (viewing already-loaded timeline/matches) vs. requires connectivity (generation, chat), and how errors are surfaced when the network is the actual cause, not a bug.
- **App Store / Play Store submission realities** — both platforms have review processes, content-guideline compliance (relevant here: AI-generated content disclosure, data-collection disclosure for a product handling teen data — COPPA/FERPA-adjacent considerations given the target users are largely minors), and app-store-specific onboarding UX conventions (e.g. sign-in-with-Apple is often required if any other third-party sign-in exists). Budget real calendar time for review/rejection cycles, not just build time.
- **Feature parity vs. mobile-first cuts** — decide deliberately whether v1 of the native app ships full feature parity (all 8 features + counselor tools) or a focused subset (e.g. timeline + matches + scholarships + push nudges first, essay/narrative/interview tools staying web-only initially since they're heavier, text-input-dense flows that may work fine on mobile web without needing native wrapping).
- **Analytics/crash reporting** — a native app needs its own crash reporting and usage analytics (e.g. Sentry + a mobile analytics SDK) distinct from whatever web analytics exists today (ties into 6a's eval/observability theme, extended to a new platform).

### 9. Beating the incumbents specifically (Scoir, Naviance) — Jul 21

Naviance is the 40%-of-US-high-schools incumbent — old, clunky, but entrenched with historical data no newcomer has by default. Scoir is the modern, better-UX challenger already eating Naviance's share. Kairos needs to credibly beat *both*: out-modernize Naviance, out-substance Scoir. #### 9a. Scattergrams — dropped Jul 22, product decision
**Dropped, reconfirmed Jul 23.** Explicit product call: this needs historical per-high-school outcome data Kairos doesn't have and can't reasonably acquire right now (see the bootstrapping problem below), so it's off the roadmap rather than left as an aspirational open item. No code was ever built for this (idea-stage only, unlike war room above), so there's nothing to remove — this entry stays as a record of why it was passed on, not as active backlog.
Naviance's scattergram (GPA on Y, SAT/ACT on X, plotted against *your specific high school's* historical accepted/waitlisted/denied applicants to a given college) is the feature counselors and students cite most when defending Naviance despite its clunkiness — it's real, local, high-school-specific outcome data, not a generic national admit-rate number. Kairos's current "odds"/match-tier reasoning (per Section 1's "why this match" citations) is model-reasoned, not this kind of empirical per-school historical record. This is a genuine, non-trivial gap:
- **Requires historical outcome data Kairos doesn't have yet** — this only becomes possible once enough students have gone through Kairos and self-reported (or a counselor logged) their actual admit results per school. Bootstrapping problem: needs either (a) a critical mass of real Kairos users over multiple app cycles, or (b) a data-sharing/import path from schools' existing Naviance/Scoir historical records during a school's migration onto Kairos, or (c) an anonymized cross-school aggregate model (less precise than per-high-school data, but usable from day one).
- **Design opportunity, not just parity** — a modern scattergram could go further than Naviance's static plot: hover to see the specific factors Kairos's AI weighed for that student (tying into the existing "Based on…" sourcing pattern), or blend the empirical scatter with the AI's qualitative reasoning instead of showing two disconnected numbers.
- **This is the concrete deliverable behind the "chance"/odds work already flagged** in Sections 5b/6a — flagging here explicitly since it's the single feature most likely to make a skeptical counselor or parent take Kairos seriously as a Naviance replacement, not just an add-on tool.

### 17. Admissions-officer feedback loop — flagged as a hard goal, scoped down to what's actually buildable (Jul 21)

Raised Jul 21: could counselors get real feedback from college admissions officers through Kairos. **Correctly flagged as hard** — worth being explicit about why, so it's not silently dropped but also not naively scoped as a normal feature:
- Admissions offices don't take unsolicited per-applicant feedback requests from third-party platforms, and FERPA cuts both ways — a college generally won't discuss an individual applicant's file with anyone but the counselor/student of record, and won't do it through an unaffiliated tool regardless.
- What colleges *are* sometimes willing to share is institutional/aggregate (applicant pool trends, general profile of admits), not per-student feedback — which is closer to what the scattergram work (9a) already approximates empirically than to a real conversation.
- Any real version of this is a multi-year trust/BD relationship with individual admissions offices, not a software feature — out of scope for engineering work on this timeline, belongs on a business-development roadmap instead if pursued at all.

**What actually gets at the same underlying need — a counselor wanting real signal on how a specific student will read to admissions — without requiring colleges to participate:**
- **Push the scattergram/empirical-outcomes work (9a) further as the honest substitute** — real historical outcome data (once Kairos has enough of it, or imports a school's existing Naviance/Scoir history) is the closest thing to "the feedback a college would give," without needing a college to say anything at all.
- [x] **A counselor-facing "how this application reads" AI review — shipped Jul 23.** New `APPLICATION_PACKAGING_REVIEW_PROMPT` (`lib/anthropic.ts`) and `api/counselor/application-review/route.ts` (POST): pulls a consenting student's narrative throughline/core values/differentiator, last 10 essay-feedback entries (labels/dimensions only, not full essay text), activities, and intended major, and asks the model whether the throughline reads consistently across them, naming inconsistencies and giving an overall packaging read. Consent-gated the same way as the existing "Narrative & Essays" tab (`profiles.share_narrative_with_counselor`) — checked explicitly server-side (not just relying on RLS) so a non-consenting student's counselor gets a clear "student hasn't shared this" message rather than a broken or empty feature; the client only shows the "Get AI packaging review" button when that flag is true. Rendered as a new `ApplicationReviewSection` atop `StudentDetailClient.tsx`'s existing Narrative & Essays tab. Typecheck and lint clean. Not click-tested (no counselor test account this session).
- **Aggregate admissions-officer perspective via public information colleges already publish** — Common Data Set filings, official admissions blogs, and published reader-rubric information (several schools publish what their readers weigh) could be synthesized into general "what this type of school's readers tend to weigh" guidance, distinct from and clearly labeled as *not* individualized feedback from that school — an honest, buildable version that avoids ever implying a real admissions officer reviewed this specific student.
- **Alumni/near-peer feedback network, as a human-in-the-loop alternative** — recent admits or students-turned-mentors (Kairos already has a `mentors` surface) reviewing applications is a legitimately buildable version of "an outside, informed perspective," even though it's not literally an admissions officer — worth being honest in product messaging that this is a substitute, not the real thing, to avoid the same kind of trust-gap backlash the AI-detection fear (Section 1) already showed this audience is alert to.
- **Bottom line for the list**: keep the underlying need ("give counselors/students a credible outside read before submission") active in the backlog, but implement it via 9a/mentors/rubric-review, not via a literal admissions-officer channel — flag the literal version as a future BD initiative, not an engineering task, so it doesn't quietly rot on this list as an unbuildable item.
