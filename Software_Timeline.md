### 4. Outcome data flywheel
**Complete, but a real bug was found and fixed here Jul 15 during Section 10 demo-data seeding.** Outcome-capture logging shipped correctly. The cohort view ("students like you") did not: it was marked done and calendar-gated behind a real decision cycle, but seeding 6 real `application_outcomes` rows for one school and testing it live showed "not enough data yet" regardless — the feature had never actually worked, calendar-gating just hid that. Two independent bugs in `src/lib/cohort.ts`'s `getCohortStats()`, both fixed:
  1. **Query always errored.** The nested embed `application_outcomes → school_matches!inner → profiles!inner` assumed a foreign key between `school_matches` and `profiles`; no such FK exists (both reference `auth.users` independently), so PostgREST rejected the query every time (`PGRST200`) and the function's `if (error || !data) return null` silently ate the failure. Rewrote as two flat queries (outcomes+school_matches via the real FK, then profiles by user_id) joined in JS — same anonymity guarantee, actually executes.
  2. **RLS would have hidden the data even if the query worked.** `application_outcomes` has `using (auth.uid() = user_id)` — a session-scoped client can only ever see the *caller's own* outcome row, never enough rows from other students to cross `MIN_COHORT_SIZE` (5). Switched this function to the service-role client. Safe specifically because the function only ever returns aggregate counts, never a row or `user_id`, back to its caller.
  Verified live: `/schools/[id]`'s Outcomes tab now shows real aggregate stats (Accepted 67% / 4 of 6, Rejected 17% / 1 of 6, Waitlisted 17% / 1 of 6) instead of the permanent "not enough data" state. `tsc --noEmit` and `npm run lint` clean.

The route's own code comments describe it as public/no-auth-cookie, but the middleware redirected every unauthenticated request to `/login` before the route handler ever ran — meaning the parent war-room feature could never actually be used by a parent (who has no account). Added `/api/war-room/parent/` to the prefix list; verified locally that the route now reaches its handler and correctly 404s on bad/nonexistent tokens instead of redirecting.
